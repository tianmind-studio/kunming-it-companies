# Roadmap

昆明技术机会雷达 / Kunming Tech Radar 的路线图只列可以落地、可以复核、不会牺牲数据可信度的任务。

## Phase 1: 数据可信度阶段

- [x] 明确 `data/companies.json` 是公司主数据源。
- [x] 生成 `COMPANIES.md` 与 `data/companies.csv`，避免多份公司正文手工漂移。
- [x] 为公司记录保留 `source_url`、`source_type`、`verification_status`、`last_checked`、`confidence_score` 等可信度字段。
- [x] 建立公开来源、隐私边界和机会提示规则。
- [x] 生成数据质量报告，展示总数、强来源、待复核、缺区县、弱来源和核验日期分布。
- [ ] 优先补强 `community_pending` 和低可信记录的官网、官方页或第二公开来源。
- [ ] 为缺区县记录补充公开可验证的区县信息。
- [ ] 建立更清晰的来源升级记录：每次从弱来源升级到官网/官方页时写明依据。

## Phase 2: 页面体验阶段

- [x] 用纯 HTML/CSS/JS 构建静态站，支持 GitHub Pages 和国内静态部署。
- [x] 首页展示项目定位、数据状态、搜索入口、下载入口和提交线索入口。
- [x] 支持关键词、方向、区县、核验状态、适合人群、机会提示筛选。
- [x] 公司卡片展示名称、方向、区域、核验状态、可信度、来源类型、来源链接和最后核验日期。
- [x] 展示公司之外的来源种子、社群、活动和政府项目入口。
- [ ] 增加“最近新增 / 最近复核”视图，但只基于真实 `last_checked` 或提交记录。
- [ ] 根据移动端真实访问反馈继续压缩首屏信息密度。

## Phase 3: 社区协作阶段

- [x] 建立 Add company、Update company、Report outdated info、Source lead、Community/Event 等 Issue 模板。
- [x] 建立 Pull Request 模板，要求说明来源、生成脚本和验证结果。
- [x] 建立 Support / 更正 / 移除 / 复用说明。
- [x] 提供不会 GitHub 的提交路径。
- [ ] 每周或每两周集中处理过期链接、重复记录和高优先级待复核记录。
- [ ] 公开记录重大删除、隐私更正和来源升级原因。
- [ ] 根据真实贡献者反馈简化模板字段。

## Phase 4: 自动化校验阶段

- [x] `npm run validate` 覆盖公司数据、生成产物、CSV、数据质量报告、站点结构、公开文档边界和服务器脚本语法。
- [x] GitHub Actions 在 push / pull_request 上运行最小 CI。
- [x] 校验动态链接协议，禁止数据字段混入私人联系方式和不可核验确定性说法。
- [ ] 增加链接可达性抽样检查，默认不阻塞 PR，只生成维护报告。
- [ ] 增加数据变更摘要脚本，列出新增、删除、升级来源和降级待复核记录。
- [ ] 增加重复候选检测：公司名近似、官网相同、来源相同但 ID 不同。

## Phase 5: 地方技术生态分析阶段

- [ ] 基于现有数据生成按区县、方向、来源强度的中性分析图表。
- [ ] 做公开来源覆盖缺口分析：哪些方向只有招聘平台，哪些方向缺少官方来源。
- [ ] 汇总高校、园区、活动、政府项目入口，形成可持续复核清单。
- [ ] 输出季度数据质量小结：新增记录、升级来源、删除/更正、仍需补强的方向。
- [ ] 在不收集私人信息的前提下，探索更多本地公开来源入口。

## 不做什么

- 不做“保证就业”“内部机会”“资源保真”等承诺。
- 不做公司排名、黑名单、避坑榜或负面爆料库。
- 不抓取或公开私人手机号、私人微信、聊天截图和非授权资料。
- 不把招聘平台搜索页、政府门户或活动平台入口写成确定岗位、项目或合作机会。
- 不为了页面效果引入难维护的大框架。
